# INF-TIMESTAMPER 実装計画

`docs/要件.md` を権威ある正本として、C# / .NET 8 / WPF による実装をフェーズに分割した計画書。

**役割**: Claude Code セッションが切れても次セッションが文脈を引き継げるようにする、進捗と「次にやること」の永続化。各フェーズの完了基準（DoD）を明文化する。

進捗を更新する際は、各フェーズの DoD のチェックを埋め、必要なら新しい派生タスクを追記する。

## 全体方針

- 要件.md と本ファイルが食い違った場合、**要件.md を優先**する（本ファイルは作業計画であり、仕様の正本ではない）。
- フェーズ境界では「ビルドが通り、テストが通り、最低限の実機確認ができる」状態を維持する。
- 大きな抽象化は使用箇所が出てから導入する（YAGNI）。例：「ゲーム抽象化」も MVP では INFINITAS 専用で書き、2 つ目のゲーム追加時に抽出する。

## プロジェクト構成

```
inf-timestamper/
├── InfTimestamper.sln
├── src/
│   ├── InfTimestamper/                  WPF アプリ本体（UI / ViewModel / DI 組立）
│   └── InfTimestamper.Core/             ドメイン層（UI 非依存。データモデル、状態機械、永続化、画像認識、OBS 抽象）
└── tests/
    └── InfTimestamper.Core.Tests/       xUnit テスト
```

- WPF プロジェクトは Core を参照する。Core は WPF を参照しない。
- 画像認識・OBS 接続・JSON 永続化は Core に置き、UI からはサービス境界経由で呼ぶ（テスタビリティと 2 台 PC 構成オプションの両立）。
- ViewModel は WPF プロジェクトに置く（`System.ComponentModel.INotifyPropertyChanged` の参照範囲を WPF 側に閉じる方が後々楽）。状態射影ロジックなど純粋計算は Core にも分離可能。

### 現在のフェーズ

Phase 4 サブフェーズ A+B（認識コア）完了。次は Phase 4 サブフェーズ C（検知サイクル＋デバッグ画像保存）。

Phase 4 A+B で確定した認識コアの構造:
- `ImageNormalizer`: `byte[]` PNG → 1920×1080 `Mat`。1920×1080 はクローン、1360×1080 等は左右パディング、それ以外はアスペクト比保持リサイズ＋黒帯レターボックス。
- `ImageHasher` / `HashMatcher`: aHash/pHash + ハミング距離。`HashCandidate(name, hash, threshold)` の集合に対し最近接候補を返す。
- `HashResource` / `RoiResource`: JSON ローダ。`[x,y,w,h]` 配列は `RoiJsonConverter`、`"0xff..."` hex 文字列は `HexUlongJsonConverter` で復号。空ファイルでも空リソースを返してビルドが落ちないようにしている。
- `TesseractOcrService`: tessdata 未配置時は IsAvailable=false で OCR を無効化（配信中ポップアップを避ける既存方針に沿う）。
- `SongRepository` / `SongTitleMatcher` / `LevenshteinDistance` / `TitleNormalizer`: 1808 曲の DB を Levenshtein fuzzy match。要件のしきい値ロジック (`min(3, ⌈len×0.3⌉)`) と Confirmed / Candidate / Unmatched の 3 段階。
- `songs.json` 再生成: `tools/generate_songs_json.ps1`（IIDX-Data-Table の textage/title.json と chart-info.json をマージ）。

Phase 4 サブフェーズ C 着手前の注意:
- ハッシュデータ未整備でも `HashResource.IsEmpty` で安全側に倒せる構造になっているので、検知サイクル実装はリソース有無に依らず進められる
- 状態判定は `HashResource.States` の各 state（song_select / play_start / result）に対して、すべてのサブパターンを試して最小距離の state を採用する設計を踏襲
- エッジ検知（要件 L455-463）: 「→ プレイ開始」「→ プレイリザルト」エッジでだけタイムスタンプ操作。同一状態の連続フレームは保持中の選曲情報を OCR で更新するノーオペ
- デバッグ画像保存は Debug ログレベル時のみ有効化。ファイル名衝突防止のため `{yyyyMMdd_HHmmss_fff}.png`、1000 枚超で古い順から削除
