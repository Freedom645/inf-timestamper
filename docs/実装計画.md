# INF-TIMESTAMPER 実装計画

`docs/要件.md` を権威ある正本として、C# / .NET 8 / WPF による実装をフェーズに分割した計画書。

**役割**: Claude Code セッションが切れても次セッションが文脈を引き継げるようにする、進捗と「次にやること」の永続化。各フェーズの完了基準（DoD）を明文化する。

進捗を更新する際は、各フェーズの DoD のチェックを埋め、必要なら新しい派生タスクを追記する。

## 全体方針

- 要件.md と本ファイルが食い違った場合、**要件.md を優先**する（本ファイルは作業計画であり、仕様の正本ではない）。
- フェーズ境界では「ビルドが通り、テストが通り、最低限の実機確認ができる」状態を維持する。
- 大きな抽象化は使用箇所が出てから導入する（YAGNI）。例：「ゲーム抽象化」も MVP では INFINITAS 専用で書き、2 つ目のゲーム追加時に抽出する。

## プロジェクト構成

```
inf-timestamper/
├── InfTimestamper.sln
├── src/
│   ├── InfTimestamper/                  WPF アプリ本体（UI / ViewModel / DI 組立）
│   └── InfTimestamper.Core/             ドメイン層（UI 非依存。データモデル、状態機械、永続化、画像認識、OBS 抽象）
└── tests/
    └── InfTimestamper.Core.Tests/       xUnit テスト
```

- WPF プロジェクトは Core を参照する。Core は WPF を参照しない。
- 画像認識・OBS 接続・JSON 永続化は Core に置き、UI からはサービス境界経由で呼ぶ（テスタビリティと 2 台 PC 構成オプションの両立）。
- ViewModel は WPF プロジェクトに置く（`System.ComponentModel.INotifyPropertyChanged` の参照範囲を WPF 側に閉じる方が後々楽）。状態射影ロジックなど純粋計算は Core にも分離可能。

### 現在のフェーズ

Phase 3 完了。次は Phase 4（画像認識）。

Phase 3 で確定した接続層の構造:
- `IObsConnection`: 抽象。`ObsConnectionOptions` で接続情報、`ObsScreenshot` で結果、`ObsStreamState` でイベント射影。
- `ObsWebSocketConnection`: 同期 API の OBSWebsocketDotNet をラップ。認証完了を `Connected` イベント＋`TaskCompletionSource` で待つ実装に注意（`ConnectAsync` は void 戻り）。`GetSourceScreenshot` は `data:image/png;base64,...` 形式の data URL を返すため `DecodePngDataUrl` で復号。
- `ObsConnectionManager`: 接続→切断検知→バックオフ再接続を 1 ループにまとめた。`IDelayProvider` でテスト時に時間ジャンプ可能。
- `ObsConnectionPool`: 1 台 PC 構成（`Stream` と `Capture` が同一インスタンス）と 2 台 PC 構成（別インスタンス）の切替え。2 台モードで同一インスタンスを誤って渡したら例外。
- `ObsScreenshotCapture`: `PeriodicTimer` 1Hz。`CaptureFailed` / `ScreenshotCaptured` イベントで上位（後の状態機械や認識サービス）に渡す。

Phase 4 着手前の注意:
- 取得した `ObsScreenshot.PngBytes` を OpenCvSharp で `Mat` に変換し、1920×1080 に正規化する箇所が Phase 4 の起点
- 参照画像（hashes.json）と ROI 定義（rois.json）は `Resources/INFINITAS/` 配下に手作業で用意する必要あり（image_recognition_decision.md 参照）
- `Tesseract.NET` の tessdata 同梱方法は Phase 4 で確定（exe 同階層に配置 → `csproj` の `Content` 指定）
